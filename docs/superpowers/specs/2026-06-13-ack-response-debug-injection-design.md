# コマンド ACK 応答デバッグ注入（遅延・無応答・強制エラー）設計

- 日付: 2026-06-13
- 対象仕様: `docs/specs/midi-mapping.md`（2026-06-12 版。**仕様書は変更しない** — 本機能はシミュレータ内部のデバッグ機能であり、ワイヤ仕様に新要素を追加しない）
- 基盤設計: `2026-06-12-cc-remap-slider-and-12-buttons-design.md` / `2026-06-10-cc-band-and-opcode-redesign-design.md`

## 背景

ゲーム側（Unity）はコマンド送信後、応答（CMDRSP_STATUS CC#90）を最大 30 フレーム待ち、
超過で失敗扱い・再送可、タイムアウト後に遅れて届いた応答は破棄する（仕様 §5 送受信規約）。
この**要求側の異常系**——

1. 純粋なタイムアウト → 失敗扱い → 再送（ACK が一切来ない）
2. タイムアウト後の遅延応答の破棄（ACK が 30 フレーム超過後に届く）
3. NG status（UNKNOWN_OP / INVALID_ARG / REJECTED）の処理
4. 予約帯 4–63 の未知 status を NG として扱う規約

——を対向試験するには、応答者であるコントローラ役（本シミュレータ）が ACK の**タイミング**と
**status** を意図的に操作できる必要がある。現状は受信即時に「検証 → ACK → 実行」しており、
操作手段がない。

要求（ブレインストーミングで確定）:

- ACK の応答時間を変化させるデバッグ機能（タイムアウト確認用）
- ACK を一切返さない無応答モード
- エラー status を強制送信する設定（定義済み NG 3 種＋予約値 63）

## 設計判断

### 1. 注入は Messaging（プロトコル層）に内蔵する

ACK のタイミング・status・seqEcho はプロトコル層の責務であり、タイムアウト計数（`tick()`）と
同じ駆動で遅延を数えるのが最も単純。代替案は以下の理由で不採用:

- **send_cc ラッパー方式**（CC90 送信だけを横取りして遅延）: 実行が即時に走り ACK だけ遅れる
  ため「ACK 送信後に実行」の仕様順序が崩れる（例: SetPreset の CC117 新値通知が ACK より先に
  届く不自然なワイヤになる）。強制エラー時に実行を抑止できない。
- **受信遅延方式**（midi_io 側で受信 CC を遅らせる）: 「応答時間」ではなく「受信遅延」の模擬に
  意味がずれ、強制エラーを表現できない。

### 2. 意味論: 遅延は「コマンド処理全体の遅延」、無応答・強制エラーは「実行しない」

- **遅延 N Tick**: commit から N Tick 後に「検証 → ACK → 実行」を一体で行う（仕様の処理順を
  保存）。検証も発火時＝ACK 時点の状態に対して行う（保留中に他コマンドやキー操作で状態が
  変わっても ACK と実行が整合する）。
- **無応答**: ACK も実行もしない（コマンドを受信しなかった相当）。ARG は規約どおり commit で
  消費する。受信記録には「無応答」と明示する。
- **強制エラー**: 検証結果を上書きして NG を返し、**実行しない**（ワイヤ上の応答と内部状態の
  整合を保つ。Unity に失敗と伝えたのに Preset が変わり CC117 が飛ぶ、という乖離を防ぐ）。
  OK(0) の強制は禁止（検証 NG コマンドの実行を構造的に防ぐ）。
- **ARG1/ARG2 は commit 時点で束縛・消費**（次コマンドの引数と混ざらない。既存規約のまま）。
- 保留中に複数コマンドが届いた場合は **FIFO 順で発火**（追い越しなし。遅延設定を途中で
  短縮しても順序は保存）。Unity の再送試験で「旧 seq の遅延 ACK → 新 seq の ACK」が
  順番どおり流れる。
- **Reset・clear_pending は注入設定・保留 ACK に影響しない**。注入は「ワイヤ/ファーム遅延の
  模擬」でありコントローラ状態の外側（自動モードフラグと同じ操作者所有）。Reset 再送試験の
  途中で設定が解除されると試験にならないため。

### 3. UI: ランタイムキー巡回（自動デバッグ入力モード中も有効）

- **`T`**: ACK タイミング巡回 — 即時(0) → 10 Tick（タイムアウト 30 以内の体感遅延）→
  35 Tick（タイムアウト超＝遅延応答破棄の試験）→ 無応答 → 即時 …
- **`E`**: ACK 強制 status 巡回 — 通常（検証結果どおり）→ UNKNOWN_OP(1) → INVALID_ARG(2) →
  REJECTED(3) → 予約(63) → 通常 …
- コマンド受信・ACK は自動デバッグ入力モード中も生きているため、`T`/`E` は自動モード中も
  受理する（許可キーに追加）。
- 設定変更と注入の発動はコンソールへ `[ACK 注入]` プレフィックスで明示ログする。

### 4. API / 定数の置き場所

- `Messaging.set_ack_delay(delay: Optional[int])` — `0`=即時 / `N>0`=N Tick 遅延 /
  `None`=無応答。負数は `ValueError`。
- `Messaging.set_ack_forced_status(status: Optional[int])` — `None`=通常 / `1–63`=強制。
  `0`（強制 OK）と範囲外は `ValueError`。
- `Messaging.__init__` に `on_log`（省略可・既定は無音）を追加（`controller_state` と同じ
  注入パターン。HUD を持つ層が `print` を渡す）。
- `CommandRecord` を拡張: `status: Optional[int]`（無応答時 `None`）・`dropped: bool`・
  `forced: bool`（既定値付き追加のため既存利用と互換）。
- 巡回プリセットは UI 関心事として `midi_simulator.py` に置く:
  `ACK_DELAY_PRESETS = (0, 10, 35, None)` / `ACK_FORCED_STATUS_PRESETS = (None, 1, 2, 3, 63)` /
  `RESERVED_STATUS_PROBE = 63`（仕様 STATUS 予約帯 4–63 の代表値）。
- `cc_map.STATUS_NAMES`（ログ表示用の status 名称表）を追加。
